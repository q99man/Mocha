## 오늘 한 일

- 백엔드/브리지에서 `mock`, `stub`, `dummy` 성격의 분석/채점 경로를 제거했다.
- 실제 MediaPipe 브리지 응답만 허용하도록 런타임 경로를 정리했다.

- 기존에 저장된 `stub` 레퍼런스 프로필로는 업로드 채점을 진행하지 못하도록 차단했다.
- 챌린지 목록/모션 세션에서 `stub` 레퍼런스를 `referenceMotionProfileReady=true` 로 보지 않도록 수정했다.
- 브리지 `analysis.py` 에서 `os` import 누락으로 나던 500 에러를 수정했다.
- Redis가 없는 로컬 개발 환경에서 챌린지 활성/비활성 토글 시 캐시 무효화 때문에 길게 찍히던 스택트레이스를 1회 fallback 경고로 줄였다.

## 핵심 원인 정리

### 1. 점수가 이상하게 100으로 수렴하던 원인

- 브리지 요청 로그는 찍혔지만, 실제 DB에 저장된 레퍼런스 프로필이 `mediapipe-fastapi-contract-stub` 였다.
- 즉 시도 영상은 실제 브리지를 탔더라도, 비교 기준이 예전 `stub` 프로필이면 결과가 왜곡될 수 있었다.
- 확인 위치:
  - `challenge_motion_profiles.analyzer_name`
  - `challenge_motion_profiles.profile_data.extras.analysisMode`

### 2. 브리지 500 에러 원인

- `mediapipe-bridge/app/analysis.py` 하단의 모델 경로/업로드 경로 해석 함수가 `os.getenv(...)` 를 쓰는데 `import os` 가 빠져 있었다.
- 증상:
  - `NameError: name 'os' is not defined`
- 수정 완료.

### 3. 활성/비활성 토글 시 Redis 에러 원인

- 토글 자체는 정상 적용됐다.
- 마지막에 인기 챌린지 캐시 무효화가 실행되는데, 로컬 Redis(`localhost:6379`)가 없어서 `RedisConnectionException` 경고가 발생했다.
- 기능 오류는 아니고 개발 환경 캐시 fallback 문제였다.
- 이제 첫 실패 후 Redis 캐시를 비활성화하고 repository fallback만 사용하도록 조정했다.
