# Mocha Handoff 2026-04-20

## 제품 방향
- 이 프로젝트의 핵심은 `기록 서비스`가 아니라 `노래방처럼 가볍게 도전하고 반복해서 즐기는 챌린지 플레이 경험`이다.
- 사용자는 플레이 후 직관적으로 결과를 확인하고 바로 다시 한 판 더 하고 싶어야 한다.
- 기록 보관, 장기 히스토리, 상세 아카이빙은 우선순위가 낮다.
- 게시판, 마이페이지, 후기는 서비스가 너무 비어 보이지 않게 받쳐주는 보조 영역이다.
- 메인 집중 영역은 `챌린지 플레이`와 `관리자 운영 허브`다.

## 현재 우선순위
1. 챌린지 플레이와 결과창의 DJMAX 감성 고도화
2. 실제 모션 인식/평가 품질 개선
3. 관리자 운영 허브 고도화
4. 게시판/마이페이지/후기는 보조 영역으로 최소 정리

## 챌린지 플레이 원칙
- 플레이 흐름은 짧고 직관적이어야 한다.
- 화면은 게임처럼 보여야 하며, 서비스 페이지처럼 느껴지면 안 된다.
- 결과는 자세한 기록보다 `한눈에 이해되는 점수와 판정감`이 중요하다.
- 버튼은 적고, 재도전 흐름은 짧아야 한다.
- 반응형이어도 레퍼런스 영상과 판정 연출이 무너지면 안 된다.

## 현재까지 반영된 상태
- 챌린지 플레이 화면은 좌/우 분할 구조를 유지한다.
- 플레이 종료 후 `클리어 -> 분석 -> 결과` 흐름이 자연스럽게 이어지도록 정리되어 있다.
- 결과창은 보드형 화면이 아니라 게임 결과창 톤으로 유지한다.
- 점수는 카운트업 연출로 표시된다.
- 플레이 중 판정 연출은 `타임라인 기반 슬롯` 구조로 들어가 있다.
- 하단 비트 레일과 중앙 판정 팝업이 있으며, 현재는 프리뷰 판정으로 동작한다.
- 내부 구조는 이후 실제 모션 인식 결과를 슬롯에 매핑하기 쉽게 나뉘어 있다.

## 판정 시스템 메모
- 현재 파일: `frontend/src/features/challenges/playJudgement.ts`
- 핵심 구조:
  - `PlayJudgementPlanItem`: 판정 슬롯
  - `PlayJudgementEvaluation`: 슬롯에 들어갈 평가 결과
  - `PlayJudgementCue`: 화면에 뿌릴 최종 판정 데이터
- 현재는 `timeline-preview` 소스로 판정을 만들어 UI를 구동한다.
- 이후 실제 모션 인식 연결 시에는 `motion-analysis` 소스로 교체하면 된다.
- 현재 확장 가능한 판정 종류:
  - `PERFECT`
  - `GOOD`
  - `HOLD`
  - `EARLY`
  - `LATE`
  - `MISS`

## 이후 모션 인식 연결 방향
- 나중에 초 단위 혹은 더 세밀한 모션 인식 결과가 들어오면, 각 판정 슬롯에 대해 다음 정보를 계산해 넣는 구조로 간다.
- 필요한 값:
  - 실제 입력 시점 또는 매칭 시점
  - 기준 박자 대비 오프셋(ms)
  - 판정 결과(`PERFECT/GOOD/EARLY/LATE/MISS`)
  - 신뢰도 또는 유사도 점수
- 즉, `타임라인을 먼저 만들고`, `실측 결과를 해당 슬롯에 평가값으로 주입`하는 방식으로 유지한다.

## 운영 허브 방향
- 운영 허브는 단순 등록 화면이 아니라 운영자가 빠르게 보고 바로 조치하는 도구여야 한다.
- 최종적으로는 아래를 포함하는 방향으로 확장한다.
- 예정 항목:
  - 챌린지 등록/수정/분석 보기 고도화
  - 게시판 관리
  - 회원 관리
  - 상태 모니터링
  - 모델 업로드 및 운영 흐름 정리
